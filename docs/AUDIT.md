## 3. Storage model

### 3a. Markdown entity files (source of truth)

Layout: `~/.brain/entities/<type>/<slug>.md`. Frontmatter schema (created by `entities.create_entity`:47):

```yaml
---
type: <singular, e.g. "person", "project">      # entities.py:18 — irregular plurals via _IRREGULAR_SINGULAR map
name: "Canonical Name"
status: current | contested | archived | superseded
first_seen: YYYY-MM-DD
last_updated: YYYY-MM-DD
source_count: <int>                              # incremented on append (entities.py:148)
tags: []
aliases: [list]                                  # only set during merge (reconcile_merge.py:139)
# any free-form metadata from extraction.metadata is added verbatim
---
```

Fact line format: `- <fact text> (source: <session_id>, <YYYY-MM-DD>)` (`apply_extraction.py:86`). Source provenance regex: `_SOURCE_RE = r"\(source:\s*([^,)]+?)(?:,\s*([\d-]+))?\s*\)"` (`db.py:81`).

### 3b. SQLite mirror (cache, rebuildable)

Location: `~/.brain/.brain.db`. Schema (`db.py:36-79`):

| Table / index | Columns | Purpose |
|---|---|---|
| `entities` | id, path, type, slug, name, status, summary, first_seen, last_updated, source_count, tags | one row per markdown file |
| `entities_type_idx` | (type) | filter by type |
| `entities_updated_idx` | (last_updated) | powers `brain_recent` |
| `aliases` | (entity_id, alias) PK; `aliases_alias_idx` on alias | name → entity lookup |
| `facts` | id, entity_id, text, source, fact_date | one row per `- ` bullet |
| `facts_entity_idx` | (entity_id) | parent join |
| `fts_facts` (FTS5) | text, source — `content='facts'`, `content_rowid='id'`, `tokenize='porter'` | BM25 fact search (`db.py:286`) |
| `fts_entity` (FTS5) | name, aliases, summary — `content=''` (external), porter | entity-name search (`db.py:311`) |

PRAGMAs: `journal_mode=WAL`, `synchronous=NORMAL`, `temp_store=MEMORY` (`db.py:37-39`).

### 3c. Harvest ledger

Location: `~/.brain/.harvest.db` (mentioned as `??` in the `~/.brain` git status — file is untracked). Schema (`harvest_session.py:46-52`):

```
CREATE TABLE harvest_state (
  session_id  TEXT PRIMARY KEY,
  path        TEXT NOT NULL,
  byte_offset INTEGER NOT NULL DEFAULT 0,
  last_seen   REAL NOT NULL
)
```

Authoritative for "have we ingested this many bytes of this session?". Legacy `~/.brain/.harvested` text file is still maintained and rotated to 2,000 entries (`harvest_session.py:95-104`) for backward compat with `clean.clean_stale_harvested`.

### 3d. Semantic mirror

Location: `~/.brain/.vec/`. Built by `semantic.build()` (`semantic.py:82`). Files: `facts.npy`, `facts.json`, `entities.npy`, `entities.json`, `meta.json`. Currently: 3,006 facts / 897 entities embedded with `all-MiniLM-L6-v2` at 384-d (`.vec/meta.json`). Stale-threshold 6 h (`semantic.py:162`). **Not invoked by any MCP tool.**

### 3e. Identity files

Three free-form markdown files in `~/.brain/identity/`: `who-i-am.md`, `preferences.md`, `corrections.md`. Read by `brain_identity()` (`mcp_server.py:120`). Corrections are the only one with an enforced append schema (`apply_extraction._apply_corrections`:109). No graph between them, no health/finance/legal/relationship slots.

### 3f. Open-vocabulary type system

`config.ENTITY_TYPES` is discovered at import time from any subfolder of `~/.brain/entities/` (`config.py:27-36`); new types are minted on first use via `get_or_create_type_dir()` (`config.py:43-51`). The extraction prompt (`prompts/extract_session.md:38`, `prompts/extract_batch.md:24`) tells the LLM "Type is free … common ones: people, projects, domains, decisions, issues, insights, meetings, techniques, quotes, questions" but allows new ones. Holding it together: an `{existing_entities}` block fed into every prompt (`auto_extract.py:48-90`), the dedup pass in `reconcile.find_possible_duplicates` (`reconcile.py:105`), and the irregular-plural map (`entities.py:13`). What could fragment: nothing prevents the LLM from emitting `person` vs `people` vs `humans` for the same concept across runs; only the existing-names cache nudges it. Live vault has at least: `clients`, `corrections`, `decisions`, `domains`, `evolutions`, `insights`, `issues`, `people`, `projects` (per `_MOC.md` files).

## 4. Retrieval surface

| `brain_identity()` (`mcp_server.py:119`) | concat 3 files | Session-start preamble | <2 ms |
| `brain_stats()` (`mcp_server.py:134`) | `COUNT(*)` + `GROUP BY type` | Sanity check | <5 ms |
| `brain://identity` resource (`mcp_server.py:149`) | thin wrapper over `brain_identity()` | for resource-style clients | — |

**Critical gap:** `semantic.search_facts`, `semantic.search_entities`, and `semantic.hybrid_search` exist (`semantic.py:208`, `:233`, `:250`) and are exposed via the `brain.semantic` CLI but are **not registered as MCP tools**. Claude cannot do meaning-based retrieval through MCP.

### Direct SQLite query helpers

- `db.search(query, k, type)` — BM25 facts (`db.py:286`)
- `db.search_entities(query, k)` — BM25 entities (`db.py:311`)
- `db.connect()` context manager (`db.py:85`)
- `db.rebuild()` — full reindex from disk (`db.py:267`)

### Legacy "preload index.md" path

The README (lines 73-79) and `check_pending.sh` describe a model where Claude is told to "Read identity files at session start" and consult the brain. `index.md` is rebuilt by `index.rebuild_index()` (`index.py:40`) and currently spans 892 lines / ~216 KB per the doc-string in `db.py:3`. Loading this into a system prompt costs ~50 K tokens at today's size and grows linearly with entities — it is the *thing the SQLite mirror was built to replace*, but the README still references it.

## 5. Extraction quality controls

| Control | Where | What it does | Failure mode |
|---|---|---|---|
| Prefilter | `prefilter.filter_session_text` (`prefilter.py:81`) | Drops Read/Glob/Grep/Bash/LS/TodoWrite/WebFetch/WebSearch/BashOutput/KillShell/Task continuations; keeps Write/Edit headers; caps lines at 800 chars; collapses repeated tool stubs | Aggressive — loses signal if a useful fact appeared inside a `Bash` tool. Recovery: the user reverts the session with the raw `.md` still on disk pre-cleanup. |
| Batched LLM calls | `_process_batch` (`auto_extract.py:197`) | Up to `BATCH_SIZE=10` sessions per LLM call | If batch parsing fails or LLM drops a session, falls back to `_process_single` per file (`auto_extract.py:215`, `:242-244`) |
| Retry tracking | `.retries` sidecar files (`auto_extract.py:104-117`) | Up to 2 retries; 3rd LLM failure → drop the raw file | Silent data loss after 3 failures. No alert. `clean.clean_orphan_retries` (`clean.py:29`) sweeps stale markers. |
| Entity-name cache | `_build_entity_name_cache` (`auto_extract.py:48`) | Compact `## type / - name` listing fed into prompt as `{existing_entities}` so LLM reuses canonical names | Cache invalidation is heuristic — assumes `index.md` mtime is a proxy for entity changes (`auto_extract.py:82-84`). After batch run, cache is unlinked (`auto_extract.py:299`). |
| Fact dedup | `_normalize_fact` (`entities.py:88`) + `existing_facts` set (`entities.py:124`) | Strips bullet, source-suffixes, lowercases → dedup key. `append_to_entity` skips lines whose key already exists. | Case-only differences merge correctly; semantic near-duplicates do not (e.g. "Hung hardcoded 245 kW" vs "Hung set P1=245 kW" — both stored). |
| Source-suffix collapse | `clean.collapse_double_sources` (`clean.py:100`) | Collapses `(source: A) (source: B)` → keeps only the last | Already-corrupted lines fixed in next clean cycle. |
| Reconcile detector | `reconcile.find_possible_duplicates` (`reconcile.py:105`) | Three signals: word-overlap ≥50% (≥2 shared) or strict subset; `_compact()` lowercase-alphanumeric equality (`mover-os` ≡ `moveros`); Levenshtein ≤2 on slugs ≤16 chars | O(N²) over entities (per type). At 10k entities/type this is 50M comparisons — minutes per cycle. |
| Auto-merge | `reconcile_merge.is_high_confidence` (`reconcile_merge.py:43`) + `merge_pair` (`:167`) | Compact-equal → safe; `SPLITTING_WORDS` set (friday/lunch/inventory/…) and ≥2 distinct 3+ digit numbers → unsafe; subset-only merges allowed for `people` only | False merges still possible — `dana-enterprise` and `dana-enterprise-local-development` were merged (visible in current `aliases:` field). Whether that was correct depends on user intent; system can't ask. |
| LLM tiebreaker | `llm_dedup.judge_pair` (`llm_dedup.py:67`) | For pairs the heuristic skipped, asks Haiku duplicate? canonical? | Single-shot JSON parse with bracket-fallback (`llm_dedup.py:50-64`); if Haiku is wrong the merge is destructive (no per-entity history). |
| Reconcile manifest | `reconcile.prepare_reconciliation` + `write_reconciliation_file` (`reconcile.py:160-187`) | Writes `~/.brain/timeline/<date>-reconcile-<HHMM>.md` for human review | Not surfaced anywhere — `check_pending.sh:11-15` greps for "Need your decision". Decisions, when made, are not codified back into the brain automatically. |
| Stale archival | `clean.archive_stale_entities` (`clean.py:133`) | Moves `status: archived`/`superseded` files to `entities/_archive/<type>/` | Only triggers on explicit status flag — nothing sets these flags automatically. |
| MOC regeneration | `clean.generate_mocs` (`clean.py:158`) | Per-type `_MOC.md` for Obsidian browse | Quadratic-ish read of all files per cycle. |
| File lock on corrections | `_apply_corrections` uses `fcntl.flock` (`apply_extraction.py:115-142`) | Concurrent extractions can't clobber each other | Only `corrections.md` is locked; concurrent writes to entity files rely on per-process serialization (the `flock` singleton in `auto-extract.sh:18-25`). |

## 7. Architectural risks for scale

Current vault: **897 entities, 3,006 facts, 892-line index.md, ~216 KB index** (per `.vec/meta.json` and `db.py:3` doc-string).

| Mechanism | Today (≈1k entities) | At 10k entities | At 100k entities |
|---|---|---|---|
| `index.rebuild_index()` (`index.py:40`) | Reads 897 files, writes 892-line file in <1 s | ~10× files, ~10× output → **single .md ≈ 9k lines / 2 MB**; rewritten on every extract run; git commits balloon | One file >20 MB; loading into Obsidian or any LLM is impossible. Index becomes unusable. |
| `_build_entity_name_cache` + `{existing_entities}` in prompts (`auto_extract.py:48`) | ~900 lines passed to Haiku → ~3k tokens per extract call | ~30k tokens — exceeds Haiku's prompt budget for the rest of the conversation; cost per extract grows linearly | **Breaks completely.** Must switch to "fetch top-K relevant entities by embedding before prompt" |
| `_FTS_SAFE … OR " OR ".join(tokens)` (`db.py:329-337`) | Acceptable precision at 3k facts | OR'd queries match too many irrelevant facts; BM25 ranking is the only filter; precision falls | Practically useless without phrase queries / proximity / per-field weights |
| `append_to_entity()` (`entities.py:109`) | Reads + rewrites entity file on every fact append | A "people/son.md" entity that accretes 500 facts is rewritten 500 times — quadratic write cost | Severe write amplification; git churn explodes |
| `find_possible_duplicates()` (`reconcile.py:105`) | O(N²) per type = ~900^2≈800k comparisons; runs in seconds | 10k per type → 100M comparisons per cycle; minutes | Not feasible. Need blocking by type+first-letter or LSH. |
| `clean.collapse_double_sources` etc. (`clean.py:106-130`) | Walks every entity file each cron cycle | 10k file reads per cycle = ~5 s of disk I/O on SSD | 100k reads is unacceptable per cycle; needs incremental |
| `~/.brain/.git` repo | One commit per extract batch + per ingest | ~10× more commits/day; git log scales but pack files grow | git operations slow noticeably; `git status` walks 100k+ markdown files |
| MCP token budgets — `brain_get` returns full file | Avg entity is ~2 KB | Some entities (e.g. `dana-enterprise.md` already has 30+ facts) hit 10 KB+ | A 100-fact entity per `brain_get` call burns 5k+ tokens per retrieval |
| FTS5 storage | `fts_facts` content table + porter index ≈ ~1 MB at 3k facts | ~10 MB | ~100 MB — fine for SQLite, but rebuild() takes longer |
| `harvest_session.harvest_all` scan (`:296`) | Iterates every JSONL in `~/.claude/projects/`; ledger keyed by `session_id` skips already-current ones | Same — bounded by Claude Code project count, not brain size. Safe. | Safe. |
| `semantic.build()` (`semantic.py:82`) | 14.47 s for 897 entities + 3,006 facts (per `meta.json`) | ~150 s for 9× — still manageable as a daily background job | ~25 min — needs incremental embedding (one row at a time on insert) |
| `semantic.hybrid_search` brute-force (`semantic.py:195-205`) | (3000, 384) matmul = <5 ms | (30k, 384) ≈ 50 ms — borderline | (300k, 384) ≈ 500 ms per query; need HNSW / FAISS |
